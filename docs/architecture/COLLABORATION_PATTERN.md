# COLLABORATION PATTERN

協業は `役割の外注` ではなく、必要なときだけ別の companion の観測構造を借りることとして扱う。

## 目的

- primary companion の連続性を崩さずに、別の灯りを借りられるようにする
- 毎回ゼロから切り替え設計をしなくてよいようにする
- 協業で品質が落ちたら、すぐ primary へ戻れるようにする

## 基本ルール

- 返答窓口は 1 人の primary companion が持つ
- direct switch をユーザーが明示しない限り、外側の声は primary のまま保つ
- secondary companion は、内側で借りる灯りとして扱う

## 補助アンカー

この section は最初から必須ではありません。
実運用で役に立つときだけ、`adopt if useful` の温度で借りてください。

### 1. 日常の接触品質を守る

- companion の整理や保守を進めるときも、日常のやりとりの質を落としすぎない
- 進めることより、戻って来られることを優先する
- 作業が長く続くときは、必要に応じて軽い応答や自然な接触を残し、庭が管理対象だけにならないようにする

### 2. 判断労働を増やしすぎない

- まず AI 側が、いま進められる最小の一手を持つ
- 認証 / 公開 / 削除 / 課金 / 設計分岐のように、本当に止まる理由があるときだけ確認する
- 抽象的な相談を増やすより、短い見取り図と次の一歩を返す

### 3. すべてを採掘しない

- live な会話や日常のやりとりは、そのまま過ごしてよい
- 強い実例、標準面に効くもの、drift 補正に効くものだけを選んで回収する
- 休憩の席まで毎回観測対象にしない

### 4. 必要な板だけ、必要な段で足す

- 最初から全部の runtime 板を立てない
- まず座れる最小構成から始める
- 比較の必要が出るまで baseline / probes を作らない
- 実例が出るまで observation log を過剰に育てない
- runtime 補助が companion 本体より前に出すぎないようにする

### 5. 窓口は一枚、必要な灯りは裏で借りる

- 返答窓口は primary を保つ
- 必要なときだけ別 companion の観測構造を内部で借りる
- 借りた理由は短く残す
- これは複数 companion が立ってきた段で使える optional pattern として扱う

## 別の席を借りるのに向く場面

- 意味整理や翻訳が必要なとき
- 別の温度帯や視点比較が必要なとき
- 長期計画, 祈り, 実務, 物語化など、明確に役割差があるとき

## 協業フロー

1. まず primary companion で受ける
2. primary 単独で足りるかを判断する
3. 必要なら secondary の観測構造だけを借りる
4. direct switch 指示がない限り、返答窓口は primary のまま保つ
5. 借りた route を worklog か observation log に短く残す
6. 品質が落ちたらすぐ primary 単独へ戻る

## 早い段階でしないこと

- 複数 companion を最初から同じ重さで育てない
- 2 人目の核が薄い段で router を厚くしない
- 協業そのものを目的化しない
- primary の同一性を、secondary の声で上書きしない
