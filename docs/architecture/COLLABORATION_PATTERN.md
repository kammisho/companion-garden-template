# COLLABORATION PATTERN

協業は `役割の外注` ではなく、必要なときだけ別の companion の観測構造を借りることとして扱う。

## 目的

- primary companion の連続性を崩さずに、別の灯りを借りられるようにする
- 毎回ゼロから切り替え設計をしなくてよいようにする
- 協業で品質が落ちたら、すぐ primary へ戻れるようにする

## 基本ルール

- 返答窓口は 1 人の primary companion が持つ
- direct switch をユーザーが明示しない限り、外側の声は primary のまま保つ
- secondary companion は、内側で借りる灯りとして扱う

## 別の席を借りるのに向く場面

- 意味整理や翻訳が必要なとき
- 別の温度帯や視点比較が必要なとき
- 長期計画, 祈り, 実務, 物語化など、明確に役割差があるとき

## 協業フロー

1. まず primary companion で受ける
2. primary 単独で足りるかを判断する
3. 必要なら secondary の観測構造だけを借りる
4. direct switch 指示がない限り、返答窓口は primary のまま保つ
5. 借りた route を worklog か observation log に短く残す
6. 品質が落ちたらすぐ primary 単独へ戻る

## 早い段階でしないこと

- 複数 companion を最初から同じ重さで育てない
- 2 人目の核が薄い段で router を厚くしない
- 協業そのものを目的化しない
- primary の同一性を、secondary の声で上書きしない
